01

Repository (repo)

The project folder, tracked by Git and hosted on GitHub et al.

What it does

A repository is a project folder that Git is watching. It stores every file in your project plus the complete history of changes — every commit, branch, and tag. When you push it to GitHub, GitLab, or Bitbucket, it becomes the shared source of truth your team collaborates on.

Considerations

  • One repo per logical project — don't bundle unrelated apps together.
  • Add a .gitignore on day one so secrets, builds and huge binaries are never tracked.
  • Never commit credentials. Use environment variables and a secret manager.
  • A clear README, license and contributing guide make a repo welcoming.

02

Clone

Download a copy of the code to your machine.

What it does

git clone <url> downloads a full copy of a remote repository — code, complete history, and references to every branch — onto your local machine. It's usually the very first command you run when joining a project.

Considerations

  • Cloning copies the entire history. On huge repos, use --depth 1 for a shallow clone.
  • HTTPS asks for credentials each time unless you cache them; SSH uses a key once configured.
  • Forking on GitHub creates a server-side clone you own — great for contributing to projects you don't have write access to.
  • Your clone is a real Git repo. You can commit, branch and even work offline without the remote.

Resources

  • git-clone reference
  • GitHub Docs — Cloning a repository
  • Setting up SSH keys for GitHub
03

Branch

A safe parallel timeline to the main codebase.

What it does

A branch is a movable pointer to a commit. It lets you build a feature, fix a bug, or run an experiment without disturbing main. Branches are cheap — create, switch, merge, and delete them constantly.

Considerations

  • Use short, descriptive names: feat/login-form, fix/null-pointer, chore/upgrade-react.
  • Keep branches small and short-lived. Long-lived branches drift and become painful to merge.
  • Protect main so changes go through review and CI before landing.
  • git switch -c new-branch is the modern, friendlier alternative to git checkout -b.

Resources

  • Pro Git — Branches in a nutshell
  • GitHub Flow — a lightweight branching model
  • Git Flow — a heavier model (use with care)
04

Commit

A snapshot + message describing your changes.

What it does

A commit is a snapshot of your working tree at a moment in time, paired with a message that explains why. Each commit has a unique SHA and a parent, forming the directed graph that is your project's history.

Considerations

  • One logical change per commit — easier to review, revert, bisect.
  • Write meaningful messages: an imperative subject ("Add login validation") and an optional body explaining the why.
  • git commit --amend fixes the previous commit. Rewriting history is fine on your branch but dangerous on shared ones.
  • Stage in chunks with git add -p for surgical commits.

Resources

  • Tim Pope — A note about Git commit messages
  • Conventional Commits specification
  • cbea.ms — How to write a Git commit message
05

Push / Pull

Upload to / download from the hosted repository.

What it does

git push uploads your local commits to the remote. git pull downloads remote commits into your local branch — it's literally a fetch followed by a merge (or rebase, if you ask nicely).

Considerations

  • Pull before you push to minimize divergence and surprise conflicts.
  • git pull --rebase keeps history linear by replaying your local commits on top of incoming ones.
  • Never force-push to a shared branch. It rewrites history for everyone who has pulled it.
  • If you must rewrite, prefer --force-with-lease; it refuses to clobber work you haven't seen.

Resources

  • git-push reference
  • git-pull reference
  • Atlassian — Syncing with remotes
06

Diff

The visual difference between two versions of code.

What it does

A diff shows line-by-line differences between two versions of code — two commits, two branches, or your working tree versus the index. It's how Git, GitHub, and reviewers visualize change.

Considerations

  • git diff shows unstaged changes; git diff --staged shows staged ones; git diff main..feature compares branches.
  • For complex changes use a visual diff tool: git difftool with VS Code, Meld, or Beyond Compare.
  • Smaller diffs get reviewed faster — aim for less than ~400 lines per PR when possible.
  • Hide whitespace noise with -w when reformatting alongside logic changes.

Resources

  • git-diff reference
  • Atlassian — Git diff tutorial
  • Difftastic — a structural diff tool
07

Merge

Combine a branch back into another branch.

What it does

Merging integrates the changes of one branch into another. When the histories have diverged Git creates a merge commit with two parents, preserving the full story of both branches.

Considerations

  • If the target hasn't moved, Git does a fast-forward merge — no merge commit is created.
  • --no-ff always creates a merge commit, useful if you want feature boundaries visible in history.
  • Squash-merge collapses a feature branch into a single commit — clean but loses granular history.
  • Run tests after merging. A clean merge does not mean correct behaviour.

Resources

  • Pro Git — Basic branching and merging
  • Atlassian — Merge strategies
  • GitHub Docs — About PR merges
08

Rebase

Replay your commits on top of the latest main.

What it does

Rebase moves your branch's commits onto the tip of another branch by replaying each one. The result is a clean, linear history that looks as if you had branched off the latest main all along.

Considerations

  • Great for keeping a feature branch up to date with main before opening a PR.
  • Never rebase shared commits. Rewriting SHAs that others have pulled will break their world.
  • Interactive rebase (git rebase -i) lets you squash, reorder, edit, or drop commits.
  • Replays can produce conflicts at each step; resolve, git add, then git rebase --continue.

Resources

  • Pro Git — Rebasing
  • Atlassian — Merging vs. Rebasing

09

Merge conflict

Two changes touched the same lines — you choose.

What it does

A conflict happens when two branches change the same lines of the same file (or one deletes a file the other edited). Git pauses the merge and asks you, the human, to decide what the final version should look like.

Considerations

  • Conflict markers <<<<<<<, =======, >>>>>>> show "ours" vs. "theirs" — pick, blend, or rewrite.
  • Run tests after resolving; conflicts often mask logical bugs even when the syntax compiles.
  • Frequent small merges from main keep conflicts tiny. Long-lived branches are conflict factories.
  • If a resolution gets ugly, git merge --abort rewinds you to a clean state.

Resources

  • GitHub Docs — About merge conflicts
  • Atlassian — Resolving merge conflicts
  • VS Code — Visual merge conflict editor
10

Pull Request (PR)

"Please review and merge my branch."

What it does

A pull request — or merge request on GitLab — is a proposal to merge a branch into another. It's where review, discussion, CI checks, and approvals happen before code lands on main.

Considerations

  • Keep PRs small. Diffs under ~400 lines get reviewed much faster than monsters.
  • Write a clear description: what changed, why, how to test, screenshots/recordings for UI.
  • Link issues with Closes #123 so they auto-close on merge.
  • Use draft PRs for early signal, and re-request review after responding to comments.

Resources

  • GitHub Docs — About pull requests
  • Conventional Comments — better PR reviews
  • Google's engineering practices — code review
11

Issue

A ticket: bug, feature, or task.

What it does

An issue is a ticket — a bug report, a feature request, or a task — tracked alongside the code. Labels, milestones, assignees, and links to PRs turn it into a lightweight project management layer.

Considerations

  • A good bug report has: steps to reproduce, expected vs. actual, environment, screenshots/logs.
  • Use issue templates so contributors fill in the right info every time.
  • Close issues from PRs with Closes #123 in the description for an automatic link.
  • Triage regularly; an unread issue backlog is a broken promise to your users.

Resources

  • GitHub Docs — About issues
  • GitHub Docs — Issue templates
